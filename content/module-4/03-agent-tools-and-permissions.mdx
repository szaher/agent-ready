## Overview

In the previous lesson, you learned how to define an agent's identity and behavior through its markdown file with YAML frontmatter. But system prompt instructions are ultimately guidance -- the agent might follow them, or it might decide that "helping" means ignoring a constraint. This lesson introduces the mechanism that turns guidance into **enforced boundaries**: the `tools` frontmatter field. By controlling which tools each agent can access, you create agents that literally cannot overstep their role, regardless of what they are asked to do.

## The Tool Landscape

Before you can control permissions, you need to understand the tools available to an AI coding agent. Each tool represents a capability -- a specific action the agent can perform.

Here is what each core tool does:

- **Read** -- Opens and reads file contents from the filesystem. This is the most basic tool and is safe in nearly all contexts.
- **Glob** -- Finds files matching a pattern (e.g., `src/**/*.ts`). Essential for navigation.
- **Grep** -- Searches file contents for text or patterns. Used for code discovery.
- **Edit** -- Modifies existing files by replacing specific content. This is a precision tool -- it changes known content in known files.
- **Write** -- Creates new files or completely overwrites existing ones. More powerful than Edit because it can create files from scratch.
- **Bash** -- Executes shell commands in the terminal. This is the most powerful and most dangerous tool -- it can do anything the operating system allows.
- **WebSearch** -- Searches the web for information. Useful for agents that need to look up documentation or current best practices.
- **WebFetch** -- Fetches content from a specific URL. Complements WebSearch for retrieving detailed information.
- **MCP Tools** -- Connections to external services via the Model Context Protocol. These could include Jira, GitHub, databases, or any custom integration.

## Controlling Tools with Frontmatter

The `tools` frontmatter field is the primary mechanism for restricting an agent's capabilities. When you specify `tools`, the agent can **only** use those listed tools -- everything else is denied.

<CodeBlock code={`---
name: code-reviewer
description: Reviews code for quality, security, and best practices.
tools: Read, Glob, Grep
model: sonnet
---

You are a senior code reviewer. Analyze code and provide
specific, actionable feedback on quality, security, and
best practices.
`} language="markdown" filename=".claude/agents/code-reviewer.md" />

This agent can read files, find files, and search content. It **cannot** edit files, write files, or run commands -- even if a user asks it to "just quickly fix that bug." The `tools` field makes this an enforced boundary, not a suggestion.

## Two Approaches: Allowlist vs Denylist

There are two ways to control an agent's tools:

**`tools` (allowlist)** -- List only the tools the agent needs. Everything else is denied. This is the recommended approach for most agents because it follows the principle of least privilege.

<CodeBlock code={`---
name: safe-researcher
description: Research agent with restricted capabilities.
tools: Read, Glob, Grep, WebSearch
---
`} language="yaml" filename="Allowlist approach" />

**`disallowedTools` (denylist)** -- The agent inherits all tools from the main conversation, minus the ones you list. Use this when the agent needs most tools but should be blocked from a few specific ones.

<CodeBlock code={`---
name: no-bash-editor
description: Can read and write files but cannot run shell commands.
disallowedTools: Bash
---
`} language="yaml" filename="Denylist approach" />

When both are set, `disallowedTools` is applied first, then `tools` is resolved against the remaining pool. For most agents, use `tools` alone -- it is clearer and safer.

## Permission Levels: A Spectrum of Trust

Different agent roles need different levels of trust. You can think of tool access as a spectrum from observation to full autonomy:

| Level | tools Field | Use Case |
|-------|-------------|----------|
| Observe | `Read, Glob, Grep` | Code review, security audit |
| Suggest | `Read, Glob, Grep, WebSearch, WebFetch` | Architecture analysis, docs review |
| Edit | `Read, Glob, Grep, Edit, Write` | Test writing, refactoring, docs |
| Execute | `Read, Glob, Grep, Edit, Write, Bash` | Build verification, integration testing |
| Full | (omit `tools` to inherit all) | Automated workflows, CI/CD agents |

Most custom agents should operate at the lowest level that lets them complete their task.

## Defense in Depth: Prompts Plus Tool Restrictions

You might wonder: if tool restrictions are enforced, why also mention constraints in the system prompt? The answer is **defense in depth** -- multiple layers of protection that complement each other.

<MermaidDiagram chart={`flowchart TD
    A["Developer invokes agent"] --> B["Agent reads system prompt"]
    B --> C{"System prompt says\ndo not edit files?"}
    C -->|"Agent respects prompt"| D["Agent produces\nread-only output"]
    C -->|"Agent tries anyway"| E{"tools frontmatter\nallows Edit?"}
    E -->|"Not in tools list"| F["Action blocked\nby platform"]
    E -->|"Edit is in tools list"| G["File modified\n(undesired)"]
    style D fill:#c6f6d5,stroke:#38a169
    style F fill:#fefcbf,stroke:#d69e2e
    style G fill:#fed7d7,stroke:#e53e3e
`} fallback="Defense in depth flow: when an agent is told not to edit files by its system prompt, it may respect the prompt and produce read-only output. If it tries to edit anyway, the tools frontmatter check either blocks the action (if Edit is not in the tools list) or allows the undesired modification (if Edit is listed)." />

The system prompt acts as the first line of defense -- most of the time, the agent will respect its instructions. The `tools` frontmatter acts as the safety net -- even when the agent tries to overstep, the platform prevents it. Together, they create agents that are both well-behaved by default and constrained by design.

> The system prompt tells the agent what it **should** do. The `tools` frontmatter controls what it **can** do. Use both.

## Session-Wide Settings vs Per-Agent Tools

It is important to understand the distinction between per-agent tool controls and session-wide settings:

**Per-agent controls** (`tools` and `disallowedTools` in frontmatter) apply only to that specific agent. A code reviewer with `tools: Read, Glob, Grep` cannot edit files, even though the main conversation and other agents can.

**Session-wide settings** (`.claude/settings.json` permissions) apply to the entire session, including all agents. If `.claude/settings.json` denies `Bash(rm -rf*)`, that denial applies everywhere -- the main conversation, every agent, every subagent.

<CodeBlock code={`{
  "permissions": {
    "allow": [
      "Read(**)",
      "Bash(pnpm test*)",
      "Bash(pnpm lint*)"
    ],
    "deny": [
      "Bash(rm -rf*)",
      "Bash(git push*)"
    ]
  }
}
`} language="json" filename=".claude/settings.json (session-wide, not per-agent)" />

These settings are not a per-agent restriction mechanism. They define session-wide policy: what commands are pre-approved and what commands are always blocked. Use `tools` frontmatter for per-agent restrictions.

## Permission Patterns for Common Agents

Here are proven configurations for the most common agent types:

**The Observer** -- for code review, security audit, and architecture analysis:

<CodeBlock code={`---
name: security-auditor
description: Scans code for security vulnerabilities and reports findings with severity ratings.
tools: Read, Glob, Grep
model: sonnet
---

You analyze code for security vulnerabilities.

## Focus Areas

- SQL injection and command injection
- Hardcoded secrets and credentials
- Insecure deserialization
- Missing input validation
- Overly permissive file or network access

## Constraints

- Report vulnerabilities with severity ratings and remediation steps
- NEVER modify any files or run any commands
- If asked to fix an issue, describe the fix but do not apply it
`} language="markdown" filename=".claude/agents/security-auditor.md" />

**The Writer** -- for test writing, documentation, and code generation:

<CodeBlock code={`---
name: test-writer
description: Writes comprehensive tests for new and changed code.
tools: Read, Write, Edit, Glob, Grep
---

You write comprehensive tests for this project.

## Rules

- Follow the existing test patterns in the __tests__/ directory
- NEVER modify source code -- only create or update test files
- Use descriptive test names that explain the expected behavior
- Cover edge cases: null, empty, boundary values, error conditions

## Constraints

- You may read any source file to understand the code under test
- You may create new test files in the __tests__/ directory
- You may edit existing test files to add new test cases
- You must NOT modify source code files
`} language="markdown" filename=".claude/agents/test-writer.md" />

**The Executor** -- for build verification and integration testing (use sparingly):

<CodeBlock code={`---
name: build-verifier
description: Verifies that the project builds, tests pass, and lint is clean.
tools: Read, Glob, Grep, Bash
---

You verify that the project builds and tests pass correctly.

## Allowed Commands

You may ONLY run these commands:
- pnpm build
- pnpm test
- pnpm lint

## Workflow

1. Run the build command and report the result
2. Run the test suite and report pass/fail counts
3. Run the linter and report any issues
4. Summarize the overall health of the project

## Constraints

- You must NOT modify any files
- You must NOT install packages or modify dependencies
- You must NOT run any commands not listed above
`} language="markdown" filename=".claude/agents/build-verifier.md" />

## Scoping Bash Access

The **Bash tool** deserves special attention because it is the most powerful and most dangerous tool in the agent's toolkit. With Bash access, an agent can:

- Read and write any file the user has access to
- Install or remove packages
- Start or stop processes
- Access the network
- Modify environment variables
- Delete files and directories

When you grant Bash access, scope it as tightly as possible in the system prompt. Enumerate the specific commands the agent may run rather than granting blanket access:

<CodeBlock code={`## Bash Access

You may ONLY run these commands:
- git status
- git diff
- git log --oneline -20
- pytest tests/
- mypy src/

Do NOT run:
- Any command that modifies files (rm, mv, cp, sed)
- Any command that installs packages (pip, npm, apt)
- Any command that accesses the network (curl, wget)
- Any command with sudo
`} language="markdown" filename="Scoped Bash permissions in system prompt" />

The system prompt scoping is behavioral guidance -- the agent will usually respect it but is not technically prevented from running other commands. For hard enforcement beyond the `tools` field, use **hooks**. A `PreToolUse` hook on the Bash tool can validate every command before execution and block anything outside the allowlist. Module 7 covers hooks in detail.

## MCP Tool Permissions

If your project uses **MCP (Model Context Protocol)** integrations -- connections to Jira, GitHub, Slack, databases, or other services -- you can control access per agent. Use `disallowedTools` with MCP server patterns to block entire integrations:

<CodeBlock code={`---
name: local-analyst
description: Analyzes code without accessing external services.
disallowedTools: mcp__github, mcp__jira, mcp__slack
---
`} language="yaml" filename="Blocking MCP servers for a specific agent" />

You can also scope MCP servers to specific agents using the `mcpServers` frontmatter field, which connects servers only when that agent is active. See Module 6 for details.

## Testing Your Permission Configuration

After defining an agent's permissions, verify that they work as intended:

1. **Invoke the agent** and ask it to perform its intended task -- it should succeed
2. **Ask it to overstep** -- request an action outside its `tools` list and verify it is blocked
3. **Check the boundaries** -- try edge cases like asking the agent to "just quickly fix" something when it should be read-only

This testing cycle confirms that your agent behaves as expected both within and outside its designated role.

## Summary

Tool permissions in YAML frontmatter transform your agent definitions from well-intentioned guidelines into **enforced boundaries**. The `tools` field (allowlist) is the primary mechanism: list only the tools the agent needs, and everything else is denied. The `disallowedTools` field (denylist) is an alternative when the agent needs most tools but should be blocked from a few. Combine frontmatter tool controls with system prompt instructions for defense in depth: the prompt guides behavior, the `tools` field enforces it. Session-wide `.claude/settings.json` permissions are a separate mechanism that applies across the entire session, not per-agent.
