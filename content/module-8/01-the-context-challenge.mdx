Every AI coding agent operates under a constraint that most developers underestimate: the **context window**. This is the total amount of text -- code, instructions, conversation history, tool output -- that the agent can see and reason about at any given moment. It is not a soft limit. It is a hard wall. When an agent hits it, information gets dropped, earlier context gets compressed, and the quality of the agent's work degrades. This lesson explains why context is the single biggest bottleneck for agent performance, how it shapes every decision the agent makes, and what it means for how you structure your codebase.

## What Is Context, Exactly?

In everyday language, "context" means background information. For an AI coding agent, **context** has a precise technical meaning: it is everything inside the agent's working memory during a session. This includes:

- **System instructions** -- CLAUDE.md, project rules, and behavior guidelines
- **Conversation history** -- every message between you and the agent in the current session
- **File contents** -- every file the agent has read or been shown
- **Tool output** -- results from running commands, search results, test output
- **The agent's own reasoning** -- its plans, observations, and intermediate thoughts

All of this competes for space inside one fixed-size container.

The crucial insight is that everything outside the context window is **invisible** to the agent. It does not exist from the agent's perspective. A function defined in a file the agent has not read might as well not exist. A convention you mentioned twenty messages ago that has been compressed out of the window is forgotten.

## The Scale Mismatch

Here is the core problem. A modern context window is roughly 200,000 tokens -- about 150,000 words or 500 pages of text. That sounds enormous. But consider what it is up against:

Even a medium-sized project with 500 files exceeds the context window. A large monorepo can be 250 times larger than what the agent can hold in memory. The agent cannot "just read everything." It must be selective.

This mismatch is not a temporary limitation waiting to be solved by bigger models. Even as context windows grow, codebases grow faster. And there is a deeper problem: attention quality degrades with window size. An agent with 200K tokens of densely packed code performs worse than the same agent with 50K tokens of carefully curated, relevant code.

> Context is not just about capacity. It is about signal-to-noise ratio. More is not always better.

## How Agents Currently Navigate Context

Given that agents cannot read everything, they use several strategies to work within their context budget:

**Selective reading.** The agent reads files on demand rather than loading the entire codebase. It uses tools like file search, grep, and directory listing to find what it needs. Each file it reads consumes context, so it tries to read only what is relevant.

**Summarization.** As a conversation grows long, older messages get compressed. The agent or the platform automatically summarizes earlier exchanges to free up space for new information. This means details from early in a session can be lost.

**Search-first approaches.** Instead of reading files linearly, agents search for specific symbols, patterns, or filenames. This lets them jump directly to relevant code without reading everything in between.

**Context pruning.** Some platforms automatically remove tool output, long file contents, or repetitive information from the context as the session progresses.

Every one of these strategies is a trade-off. Selective reading risks missing important context. Summarization loses detail. Search can miss files that are relevant but do not match the search terms. The agent is constantly making judgment calls about what to keep and what to discard.

## The Two Failure Modes

Context problems manifest in two opposite ways, and both are equally harmful:

**Too little context.** The agent does not have enough information to do the job correctly. It misses a helper function that already exists and writes a duplicate. It does not know about a convention and violates it. It is unaware of a constraint and introduces a breaking change. This is the **starvation** failure.

**Too much context.** The agent's window is full of information, but most of it is irrelevant. A 500-line CLAUDE.md stuffed with rarely-needed details dilutes the signal. Twenty files read "just in case" push out the three files that actually matter. The agent has everything but cannot focus on anything. This is the **drowning** failure.

## Why This Matters for Your Codebase

If context is the bottleneck, then the way you organize and document your codebase directly controls how effectively agents can use their limited context budget. Every decision you make affects the agent's signal-to-noise ratio:

- **A bloated CLAUDE.md** with 500 lines of instructions consumes thousands of tokens on every session, whether those instructions are relevant to the current task or not
- **Missing documentation** forces the agent to read extra files trying to discover what you could have told it in one sentence
- **Poorly organized code** means the agent has to read more files to understand what a well-structured codebase would communicate with fewer
- **Unclear naming conventions** require the agent to read implementations to understand what a good name would have communicated instantly

> Every byte of your project documentation uses context budget. Treat context like a precious resource -- because for the agent, it is.

## The Human Analogy

Think of context like a physical desk. You can only spread out so many papers before you start losing track of things. A well-organized worker keeps only the relevant documents in front of them and files the rest where it can be found when needed. A disorganized worker piles everything on the desk and spends half their time searching through the pile.

The agent's context window is its desk. Your job as a codebase author is to make sure the right papers are easy to find and the desk is not pre-cluttered with things the agent does not need for the current task.

## What We Will Cover in This Module

Context management is a skill that spans several techniques, each building on the last:

- **Context files and kits** -- structured files that describe parts of your codebase so agents can load context on demand, not all at once
- **Memory and persistence** -- systems that carry what the agent learned about your project across sessions, so it does not re-learn the same things every time
- **Optimizing context budget** -- practical techniques for keeping your CLAUDE.md lean, your documentation high-signal, and your codebase self-describing
- **Context at scale** -- strategies for large codebases where hierarchical context, search-first approaches, and automated context selection become essential

Each of these is a lever you can pull to move your codebase from context-hostile to context-friendly.

## Summary

- The **context window** is a hard limit on everything an agent can see at once -- typically around 200K tokens
- Even medium-sized codebases exceed the context window, forcing agents to be selective about what they read
- Context problems manifest as **starvation** (too little information, leading to wrong assumptions) or **drowning** (too much irrelevant information, diluting attention)
- Every byte of documentation, every file read, and every tool output consumes context budget
- The way you structure and document your codebase directly determines how effectively agents can use their limited context
- Context management is not about making the window bigger -- it is about making every token inside the window count
